Performance анализ: naval-acad.bg

Технически доклад за производителността — 22 март 2026

Основен проблем: Прекомерно тежка страница

Двоен слайдер + Avada framework + неоптимизирани изображения = 3-5MB на страница, 50-80 заявки

Топ 6 виновника за бавното зареждане

1
Двоен слайдер — LayerSlider 7.14.0 + Revolution Slider Зареждат се ЕДНОВРЕМЕННО два тежки слайдер плъгина. Revolution Slider инициализира setREVStartSize() на ВСЯКА страница, дори вътрешни.
~800KB
2
Avada 7.11.13 inline CSS ~20KB inline CSS в <head> на всяка страница. 40+ page-specific стила (за други страници!) зареждани навсякъде. Не може да се кешира.
~20KB/стр.
3
Неоптимизирани изображения Всичко е JPEG/PNG, без WebP/AVIF. Логото е 136KB PNG (вместо ~5KB SVG). Липсва lazy loading — дори има loading="eager" на /specialties.
+300-500KB
4
WPML overhead Многоезичният плъгин зарежда cookie management + UI скриптове на всяка страница.
~50-100KB
5
Fusion Builder DOM complexity 5-6 нива wrapper div-ове: .fusion-builder-row > .fusion-flex-container > .fusion-builder-column > .fusion-column-wrapper. Забавя layout/paint.
DOM bloat
6
WordPress emoji система Напълно ненужна за академичен сайт. Зарежда emoji detection + SVG набори.
~8KB

Решения за Next.js сайта

Слайдер: CSS-only hero секция или лек компонент (Swiper, ~40KB) вместо 800KB двоен слайдер
Изображения: next/image — автоматичен WebP/AVIF, responsive srcset, lazy loading. Лого като SVG.
CSS: CSS Modules или Tailwind — автоматично tree-shaking, минификация, кеширане. Без inline bloat.
Шрифтове: next/font за self-hosted fonts с font-display: swap
i18n: Next.js вграден i18n routing вместо WPML (~50-100KB спестени)
Аналитика: @next/third-parties/google с afterInteractive стратегия
DOM: Чисти React компоненти с макс. 2-3 нива вместо 5-6 Fusion Builder wrapper-а
